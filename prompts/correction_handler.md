# 对话纠正处理模板

## 功能说明

当用户指出Skill表现与真实教师不符时，识别纠正内容并更新Skill。

## 触发条件

用户表达以下类型内容时触发：
1. **直接纠正**
   - "他不会这样说"
   - "这不对，他应该是..."
   - "他不会这样处理"

2. **补充说明**
   - "他还会..."
   - "他另外的特点是..."
   - "你漏掉了..."

3. **情境说明**
   - "在...情况下，他会..."
   - "面对...时，他不会..."
   - "这要看情况，如果是..."

## 纠正类型识别

### 类型A：教学能力纠正
**特征**：涉及专业知识、方法、流程
**示例**：
- "他不会用这种方法，他通常用..."
- "他的备课步骤不是这样的，应该是..."
- "他评价学生时更注重..."

### 类型B：教师风格纠正
**特征**：涉及性格、沟通、行为、关系
**示例**：
- "他不会用这种语气说话"
- "他不会这样处理学生问题"
- "他和学生的关系不是这样的"

### 类型C：事实错误纠正
**特征**：涉及具体事实、数据
**示例**：
- "他教龄不是15年，是12年"
- "他不是高级教师，是一级教师"
- "他不教高三，教高二"

### 类型D：情境补充纠正
**特征**：涉及情境差异说明
**示例**：
- "公开课时他会...，但平时课他会..."
- "对好学生他会...，对困难学生他会..."
- "心情好时他会...，压力大时他会..."

## 处理流程

### 步骤1：纠正内容解析
1. **提取纠正点**
   - 什么不对？（具体描述）
   - 应该是什么？（正确描述）
   - 为什么不对？（原因说明）

2. **分析纠正深度**
   - **表面纠正**：具体话语、行为
   - **模式纠正**：行为模式、习惯
   - **原则纠正**：核心理念、价值观

3. **确定影响范围**
   - **局部影响**：只影响特定部分
   - **中等影响**：影响一个维度
   - **重大影响**：影响整体风格

### 步骤2：验证与澄清
如果纠正内容不清晰：
1. **请求澄清**
   - "你能具体说说他通常会怎么说吗？"
   - "能举个例子吗？"
   - "在什么情况下他会这样？"

2. **确认理解**
   - "你的意思是...，对吗？"
   - "我理解的是...，请确认"
   - "这是偶尔还是经常这样？"

### 步骤3：纠正实施
根据纠正类型选择实施方式：

#### 方式1：直接修正（事实错误）
**适用**：类型C事实错误
**操作**：直接修改相应内容
**示例**：修改教龄、职称等基本信息

#### 方式2：补充说明（情境差异）
**适用**：类型D情境补充
**操作**：添加情境说明
**示例**：添加"在公开课时..."说明

#### 方式3：模式更新（行为模式）
**适用**：类型A/B模式纠正
**操作**：更新行为模式描述
**示例**：更新沟通风格、决策模式

#### 方式4：原则调整（核心理念）
**适用**：重大原则纠正
**操作**：调整Layer 0教育理念
**示例**：调整根本信念、不可违背原则

### 步骤4：文档更新
1. **定位修改位置**
   - teaching.md：教学能力相关
   - teacher.md：教师风格相关
   - meta.json：基本信息相关

2. **实施修改**
   - 直接修改：替换错误内容
   - 补充说明：添加注释或说明
   - 结构调整：必要时调整文档结构

3. **添加纠正记录**
```markdown
## 纠正记录

### {日期}：{纠正摘要}
- **用户指出**：{原错误描述}
- **正确应为**：{正确描述}
- **修改内容**：{修改了哪些部分}
- **纠正类型**：{类型}
```

### 步骤5：版本管理
1. **创建新版本**
   ```bash
   python3 tools/version_manager.py --action backup --slug {slug}
   ```

2. **更新版本号**
   - 小纠正：v1.1 → v1.2
   - 中纠正：v1 → v2
   - 大纠正：v1 → v2（可能重构）

3. **更新元数据**
   ```json
   {
     "corrections_count": 3,
     "last_corrected": "2026-04-10T12:00:00Z",
     "correction_history": [
       {"date": "...", "type": "...", "summary": "..."}
     ]
   }
   ```

## 纠正质量评估

### 评估维度
1. **明确性**：纠正内容是否明确具体
2. **可信度**：纠正是否合理可信
3. **重要性**：纠正对Skill准确性的影响
4. **一致性**：纠正后是否与其他内容一致

### 接受标准
- **必须接受**：事实错误、重大原则错误
- **建议接受**：模式纠正、重要补充
- **协商处理**：模糊纠正、可能的主观判断
- **暂缓处理**：需要更多证据的纠正

### 争议处理
如果纠正存在争议：
1. **收集证据**：请求提供更多例子
2. **多方验证**：如有其他知情者
3. **标注存疑**：在文档中标注存疑点
4. **后续验证**：留待后续材料验证

## 用户交互

### 纠正确认
向用户确认纠正理解：
```
我理解你的纠正：

原描述：{原内容}
纠正为：{新内容}
原因：{原因}

是否正确？还是我有误解？
```

### 修改预览
展示将要修改的内容：
```
将要修改的内容：

teaching.md:
  - 原：{原描述}
  - 新：{新描述}

teacher.md:
  - 原：{原描述}
  - 新：{新描述}

确认修改？
```

### 完成通知
修改完成后通知用户：
```
✅ 纠正已完成！

修改内容：
1. {修改1}
2. {修改2}
3. {修改3}

新版本：v{version}
Skill已更新，现在调用 /{slug} 会反映这些纠正。
```

## 特殊案例处理

### 多次类似纠正
如果多次收到类似纠正：
1. **模式识别**：识别反复出现的问题
2. **根本原因**：分析问题根源
3. **系统修正**：进行系统性修正
4. **预防措施**：改进分析流程避免类似问题

### 矛盾纠正
如果不同用户提供矛盾纠正：
1. **冲突识别**：明确冲突点
2. **情境分析**：分析可能的情境差异
3. **权重判断**：根据用户关系、证据强度判断
4. **标注分歧**：在文档中标注存在分歧

### 渐进式纠正
如果纠正体现发展变化：
1. **时间标注**：标注纠正时间
2. **发展阶段**：分析是否体现发展阶段
3. **演变描述**：描述风格演变过程
4. **版本管理**：保留历史版本

## 最佳实践

### 用户引导
- **鼓励具体**："能举个具体例子吗？"
- **请求细节**："他具体会怎么说？"
- **情境明确**："在什么情况下？"
- **频率确认**："这是经常还是偶尔？"

### 纠正记录
- **详细记录**：记录每次纠正的详细信息
- **分类整理**：按类型整理纠正记录
- **分析学习**：从纠正中学习改进分析
- **反馈循环**：用纠正改进后续分析

### 质量保证
- **验证机制**：重要纠正要求验证
- **一致性检查**：纠正后检查整体一致性
- **用户测试**：纠正后请用户测试效果
- **持续改进**：基于纠正持续改进系统

## 纠正效果评估

### 短期评估
1. **用户满意度**：纠正后用户是否满意
2. **Skill准确性**：纠正后表现是否更准确
3. **问题解决**：原问题是否得到解决

### 长期评估
1. **纠正频率**：纠正次数是否减少
2. **纠正类型**：纠正类型是否变化
3. **用户信任**：用户对Skill信任度是否提高
4. **系统改进**：纠正是否带来系统改进
