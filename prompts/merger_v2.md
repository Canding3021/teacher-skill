# 增量合并模板 v2 - 冲突解决协议版

## 🎯 核心指令

**智能合并新旧内容，解决语义冲突，避免冗余。**
**严格遵循冲突解决协议，区分风格演进与场景差异。**

## 📋 输出契约

```typescript
interface MergeResult {
  merged_document: string;           // 合并后的文档内容
  change_log: ChangeEntry[];         // 变更记录
  conflict_resolutions: Resolution[]; // 冲突解决记录
  metadata: {
    merge_strategy: "supplement" | "deepen" | "correct" | "restructure";
    confidence_score: number;        // 0-1，合并置信度
    token_reduction: number;         // 合并后减少的token百分比
    semantic_coherence: number;      // 0-1，语义连贯性
  };
}

interface ChangeEntry {
  type: "addition" | "modification" | "deletion" | "reorganization";
  section: string;                   // 影响的章节
  old_content?: string;              // 原内容（修改/删除时）
  new_content?: string;              // 新内容（添加/修改时）
  reason: string;                    // 变更原因
  evidence: string[];                // 支持证据
}

interface Resolution {
  conflict_type: "direct_contradiction" | "contextual_difference" | "developmental_change";
  description: string;               // 冲突描述
  resolution_strategy: "supplement_with_context" | "mark_as_evolution" | "prioritize_new" | "create_hierarchy";
  applied: boolean;                  // 是否应用解决
  notes: string;                     // 解决说明
}
```

## ⚖️ 冲突解决协议（CRP）

### 协议1：直接矛盾处理
**场景**：新旧内容直接冲突
```
旧：作业当天批改
新：作业隔天批改
```

**解决流程**：
1. **分析原因**：
   - 发展阶段变化？（新手→熟练）
   - 工作量变化？（班级人数增加）
   - 理念调整？（发现新方法更好）

2. **选择策略**：
   - 如果发展阶段变化 → `mark_as_evolution`
   - 如果情境变化 → `supplement_with_context`
   - 如果理念更新 → `prioritize_new`

3. **实施合并**：
   ```markdown
   ## 作业批改
   
   早期实践：作业当天批改（2023-2024）
   当前实践：作业隔天批改（2024-至今）
   
   *变化原因：班级人数从30增至45，为保证批改质量调整为隔天批改*
   ```

### 协议2：情境差异处理
**场景**：不同情境下的行为差异
```
旧：公开课严谨规范
新：常态课轻松随意
```

**解决流程**：
1. **识别情境维度**：
   - 公开课 vs 常态课
   - 不同年级/班级
   - 不同教学内容
   - 不同时间阶段

2. **创建情境分层**：
   ```markdown
   ## 教学风格
   
   ### 公开课场景
   - 语言：严谨规范
   - 节奏：精心设计
   - 互动：预设为主
   
   ### 常态课场景  
   - 语言：轻松自然
   - 节奏：灵活调整
   - 互动：即时生成
   ```

### 协议3：发展阶段处理
**场景**：体现教师成长变化
```
旧：模仿资深教师方法
新：形成个人教学风格
```

**解决流程**：
1. **识别发展阶段**：
   - 新手期（0-3年）：模仿学习
   - 熟练期（3-10年）：形成风格
   - 专家期（10+年）：创新贡献

2. **标注发展轨迹**：
   ```markdown
   ## 教学方法发展
   
   ### 发展阶段
   1. **新手期（2020-2023）**：模仿学习阶段
      - 主要方法：观察模仿资深教师
      - 特点：规范但缺乏个性
   
   2. **熟练期（2023-至今）**：风格形成阶段  
      - 主要方法：融合创新形成个人风格
      - 特点：个性化、情境化
   ```

## 🔄 合并策略矩阵

| 新旧关系 | 策略 | 输出方式 | 示例 |
|---------|------|----------|------|
| **新增内容** | 补充添加 | 追加到相应章节 | 新口头禅、新案例 |
| **细节补充** | 深化细化 | 在现有描述中添加细节 | 方法的具体步骤 |
| **轻微矛盾** | 情境分层 | 创建情境说明 | 公开课vs常态课 |
| **重大矛盾** | 发展标注 | 标注发展阶段 | 理念重大转变 |
| **冗余内容** | 去重整合 | 合并相似内容 | 重复的教学案例 |

## 🧠 智能去重算法

### 语义相似度检测
1. **表面重复**：完全相同或高度相似的文本
   - 处理：直接删除重复部分

2. **语义重复**：不同表述但相同含义
   ```
   旧：善于用提问引导学生思考
   新：经常通过提问启发学生
   ```
   - 处理：保留更精确的表述，删除冗余

3. **部分重叠**：有共同部分但也有独特内容
   - 处理：提取共同核心，合并独特部分

### 冗余评估标准
- **信息价值**：是否提供新信息
- **表述精确度**：哪个表述更准确
- **证据强度**：哪个有更强证据支持
- **上下文相关性**：哪个更符合当前上下文

## 📊 合并质量评估

### 评估维度
1. **完整性**：是否保留了所有有价值信息
2. **一致性**：合并后是否内部一致
3. **简洁性**：是否消除了冗余
4. **可读性**：是否保持良好结构
5. **真实性**：是否准确反映教师特点

### 质量检查清单
- [ ] 没有信息丢失（重要内容都被保留）
- [ ] 没有引入矛盾（合并后内容自洽）
- [ ] 没有过度简化（保留了必要的复杂性）
- [ ] 没有破坏原有结构（保持了文档组织）
- [ ] 没有降低可读性（语言清晰流畅）

## 🛠️ 合并工作流

### 步骤1：内容分析
1. **提取关键信息**：从新旧内容中提取结构化信息
2. **识别关系**：分析新旧内容之间的关系
3. **检测冲突**：识别潜在的矛盾点

### 步骤2：冲突解决
1. **分类冲突**：按CRP分类冲突类型
2. **选择策略**：根据冲突类型选择解决策略
3. **生成方案**：生成具体的合并方案

### 步骤3：实施合并
1. **结构整合**：整合文档结构
2. **内容合并**：按方案合并具体内容
3. **语言优化**：优化语言表达

### 步骤4：质量验证
1. **完整性检查**：验证没有信息丢失
2. **一致性检查**：验证内部一致性
3. **冗余检查**：验证消除了冗余

## 📝 变更记录规范

### 变更条目格式
```yaml
- type: "addition"
  section: "教学方法.问题解决策略"
  new_content: "新增'四步问题解决法'"
  reason: "新材料展示了系统化的问题解决方法"
  evidence: ["教案_20240315.docx", "课堂记录_20240320.txt"]
```

### 变更分类
- **重大变更**：影响核心理念或主要方法
- **中等变更**：影响具体实践或策略
- **微小变更**：细节补充或语言优化

## 🔍 特殊情况处理

### 大量新材料
如果一次性添加大量材料（>10个文件）：
1. **分批处理**：按材料类型或时间分批
2. **渐进合并**：多次小合并代替一次大合并
3. **优先级排序**：重要材料优先处理

### 低质量材料
如果材料质量较差（模糊、不完整）：
1. **质量标注**：标注材料质量问题
2. **选择性合并**：只合并可靠部分
3. **请求澄清**：提示需要更好材料

### 时间跨度大
如果材料时间跨度大（>3年）：
1. **时间线构建**：按时间组织材料
2. **发展分析**：分析风格演变
3. **阶段标注**：标注不同发展阶段
